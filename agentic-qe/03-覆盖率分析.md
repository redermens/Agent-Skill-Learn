# 03. 覆盖率分析 (Coverage Analysis)

## 核心 Skill: qe-coverage-analysis

**agentic-qe 的覆盖率分析能力天花板**。

### 三大分析策略

#### 1. 亚线性 Gap 检测 (O(log n))

```typescript
await coverageAnalyzer.detectGaps({
  algorithm: 'sublinear',     // O(log n) 复杂度
  source: 'src/**/*.ts',
  metrics: ['statement', 'branch', 'function'],
  sampling: {
    enabled: true,
    confidence: 0.95,
    maxSamples: 1000
  }
});
```

> **亚线性是怎么做到的**：不是真的看完所有代码，而是用统计采样 + 置信区间。对超大代码库适用。

#### 2. 风险加权覆盖率（**最有价值的概念**）

不是所有未覆盖代码都同等重要。这个模型给每段代码打风险分：

```typescript
await coverageAnalyzer.riskWeightedAnalysis({
  coverage: coverageReport,
  riskFactors: {
    complexity:       { weight: 0.30, threshold: 10 },   // 圈复杂度 >10
    changeFrequency:  { weight: 0.25, window: '90d' },   // 最近 90 天改动频率
    bugHistory:       { weight: 0.25, window: '180d' },  // 历史 bug 集中度
    criticality:      { weight: 0.20, tags: ['payment', 'auth'] }
  },
  output: {
    riskScore: true,
    prioritizedGaps: true   // 按风险排序的未覆盖列表
  }
});
```

**核心思想**: `优先级 = 未覆盖 × 风险分`

#### 3. 差分覆盖率 (Diff Coverage)

PR 级别的覆盖率检查：

```typescript
await coverageAnalyzer.diffCoverage({
  base: 'main',
  head: 'feature-branch',
  requirements: {
    newCode: 80,                // 新代码必须 80% 覆盖
    modifiedCode: 'maintain',   // 修改的代码：不能降低
    deletedCode: 'ignore'
  }
});
```

> **比"总覆盖率"更实用**。总覆盖率 80% → 85% 很难，但保证新代码 80%+ 是可达成的。

## 覆盖率阈值配置（开箱即用）

```yaml
thresholds:
  global:
    statements: 80
    branches:   75
    functions:  85
    lines:      80
  
  per_file:
    min_statements: 70
    critical_paths: 90    # 关键路径要求更严
  
  new_code:
    statements: 85
    branches:   80
  
  exceptions:                       # 豁免列表
    - path: "src/migrations/**"
      reason: "数据库迁移脚本"
    - path: "src/generated/**"
      reason: "自动生成代码"
```

**给 C++ 的建议阈值**:

```yaml
# 在 CMake / CI 配置中
thresholds:
  global:
    line:     80      # gcov 的 line coverage
    branch:   70      # branch coverage（C++ 分支多，70 已经不错）
    function: 90      # 函数级覆盖
  
  per_file:
    critical: 90      # 核心模块
  
  exceptions:
    - "tests/**"      # 测试代码本身
    - "build/**"      # 构建产物
    - "**/_generated.cpp"   # protobuf 等生成代码
    - "third_party/**"
```

## ⚠️ gcov/lcov 适配指南（重点）

### agentic-qe 完全不支持 C++ 覆盖率！

- 原生支持: Istanbul, c8, lcov.info (但只解析 JS/TS 格式)
- 不支持: gcov, gcovr, lcov 的 C++ 流程

### gcov/lcov 工作流（你需要自己搭）

```bash
# Step 1: 编译时插桩
cmake -B build -DCMAKE_BUILD_TYPE=Debug \
  -DCMAKE_CXX_FLAGS="--coverage -O0 -g"

# Step 2: 编译
cmake --build build -j

# Step 3: 运行测试（产生 .gcda 数据）
ctest --test-dir build

# Step 4: gcov 处理（产生 .gcov 报告）
cd build
find . -name "*.gcda" -exec gcov {} \;

# Step 5: lcov 聚合
lcov --capture --directory . --output-file coverage.info
lcov --remove coverage.info '/usr/*' '*/tests/*' --output-file coverage.info

# Step 6: 生成 HTML 报告
genhtml coverage.info --output-directory coverage_html

# Step 7: 提取关键指标
lcov --summary coverage.info
# Lines......: 85.3% (1234 of 1447 lines)
# Functions..: 90.2% (456 of 505 functions)
# Branches...: 72.1% (892 of 1237 branches)
```

### 让 AI 助手帮你分析 gcov 报告

虽然 agentic-qe 没有 gcov agent，但可以这样让 AI 复用 qe-coverage-analysis 的方法论：

```
prompt 示例：

我有以下 lcov 输出（粘贴 coverage.info 关键段或 lcov --list 输出）。

请按 qe-coverage-analysis 的方法分析：

1. 应用风险加权模型，对未覆盖行打分：
   - 圈复杂度高的代码权重 0.3
   - 最近改动频繁的（看 git log）权重 0.25
   - 业务关键路径（带 // CRITICAL 注释的）权重 0.2

2. 输出风险排序的未覆盖列表（前 10 条）

3. 对前 5 条，建议具体的 gtest 用例怎么写

4. 区分新代码覆盖率 vs 整体覆盖率
   （新代码 = git diff origin/main..HEAD 中的行）
```

### lcov.info 格式速查

```
SF:src/order_service.cpp
FN:10,Order::AddItem        # 函数定义在第 10 行
FNDA:5,Order::AddItem       # 被调用 5 次
DA:10,5                      # 第 10 行，命中 5 次
DA:11,5
DA:12,0                      # 第 12 行未命中 ← 未覆盖
BRDA:15,0,0,3                # 第 15 行分支 0 命中 3 次
BRDA:15,0,1,0                # 第 15 行分支 1 未命中 ← 未覆盖分支
end_of_record
```

让 AI 读这个格式很简单，方法论可以完全复用 agentic-qe 的设计。

## 完整的覆盖率分析结果接口

```typescript
interface CoverageAnalysis {
  summary: {
    statements: { covered: number; total: number; percentage: number };
    branches:   { covered: number; total: number; percentage: number };
    functions:  { covered: number; total: number; percentage: number };
  };
  
  gaps: {
    file: string;
    uncoveredLines: number[];
    uncoveredBranches: BranchInfo[];
    riskScore: number;           // 0-100
    suggestedTests: string[];    // AI 建议的测试方向
  }[];
  
  trends: {
    period: string;
    coverageChange: number;       // 趋势变化
    newGaps: number;
    closedGaps: number;
  };
  
  recommendations: {
    priority: 'critical' | 'high' | 'medium' | 'low';
    file: string;
    action: string;               // "在 OrderService::ApplyDiscount 加边界测试"
    expectedImpact: number;       // 覆盖率提升预估
  }[];
}
```

这是个**通用结构**，gcov 数据完全可以转成这个 JSON。

## 质量门禁配置

```yaml
quality_gates:
  coverage:
    block_merge:                          # 阻止合并
      - new_code_coverage < 80
      - coverage_regression > 5           # 覆盖率退步 5%
      - critical_path_uncovered           # 关键路径未覆盖
    
    warn:                                 # 警告
      - overall_coverage < 75
      - branch_coverage < 70
    
    metrics:
      track_trends: true
      alert_on_decline: 3                 # 连续 3 个 PR 下降则报警
```

## ⚠️ 关键告警（来自原文）

1. **高行覆盖 ≠ 好测试** —— 100% 覆盖 + 0% 断言很常见。配合**变异测试**验证
2. **覆盖率分析域有 14% 失败率** —— 总有 fallback 计划（手动工具）
3. **自学习管道可能静默停止** —— 状态栏卡死几天才被发现，需要人工巡检

## 历史趋势追踪

原文档建议在 `run-history.json` 持续追加：

```json
{
  "runs": [
    {"date": "2026-06-01", "statements_pct": 85, "branches_pct": 72, "gaps_found": 23},
    {"date": "2026-06-02", "statements_pct": 84, "branches_pct": 71, "gaps_found": 26},
    {"date": "2026-06-03", "statements_pct": 82, "branches_pct": 70, "gaps_found": 31}
  ]
}
```

**连续 3 次下降 = 报警**。可以让 CI 自动维护这个文件。

## 给 C++ 项目的覆盖率落地清单

1. ✅ **CMake 集成 `--coverage`**（GCC/Clang）或 `OpenCppCoverage`（MSVC）
2. ✅ **CI 跑完测试自动 lcov 聚合**
3. ✅ **HTML 报告作为 CI artifact 上传**
4. ✅ **codecov / sonarqube 上传** 持续追踪
5. ✅ **PR 级 diff coverage** 检查（用 `diff-cover` 工具）
6. ✅ **关键模块设独立阈值**（如 // COV: 95%）
7. ⚠️ **风险加权模型** 让 AI 帮你跑（不要平均化所有 gap）
8. 💡 **变异测试** 用 [mull-cxx](https://github.com/mull-project/mull) 验证测试质量
