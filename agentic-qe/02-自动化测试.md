# 02. 自动化测试 (Test Automation)

## 核心 Skill: qe-test-execution

负责测试套件的**并行执行、智能重试、实时上报**。

### 执行策略

#### 1. 并行执行

```typescript
await testExecutor.runParallel({
  suites: ['unit', 'integration'],
  workers: 4,
  distribution: 'by-file',   // 或 'by-test', 'by-duration'
  isolation: 'process',
  sharding: {
    enabled: true,
    total: 4,
    index: process.env.SHARD_INDEX
  }
});
```

**C++/gtest 等价**:

```bash
# gtest 自身的并行（每个测试一个进程）
ctest -j 4 --output-on-failure

# 或用 gtest-parallel
gtest-parallel ./my_test --workers=4

# CI 分片
ctest -j 4 -I ${SHARD_INDEX},${SHARD_TOTAL},${SHARD_TOTAL}
```

#### 2. 智能测试选择 (Affected Tests)

只跑被 PR 影响的测试，省时间。

```typescript
await testExecutor.runAffected({
  changes: gitChanges,
  selection: {
    direct: true,      // 直接修改文件的测试
    transitive: true,  // 依赖修改文件的测试
    integration: true  // 涉及修改代码的集成测试
  },
  fallback: 'full-suite'  // 分析失败时全跑
});
```

**C++ 适配思路**:

```bash
# Step 1: 拿到改动文件
git diff --name-only origin/main..HEAD > changed.txt

# Step 2: 基于 CMake 的依赖图，反查哪些 test target 受影响
# 可以用 cmake --graphviz 生成依赖图

# Step 3: 只跑相关测试
ctest -R "影响的测试名称模式"
```

#### 3. Flaky 测试处理

```typescript
await testExecutor.handleFlaky({
  detection: {
    enabled: true,
    threshold: 0.1,    // 10% 失败率即认为 flaky
    window: 100        // 最近 100 次运行
  },
  strategy: {
    retry: 3,
    quarantine: true,  // 隔离到独立 suite
    notify: ['#flaky-tests']
  }
});
```

详见 `08-Flaky测试治理.md`。

## 核心 Skill: test-automation-strategy

### 反模式识别表（**必看**）

| 反模式 | 症状 | 修复方案 |
|--------|------|---------|
| **冰激凌锥** | 80% E2E, 10% 单元 | 倒过来变金字塔 |
| **慢套件** | CI 跑 30+ 分钟 | 并行化 + 剪掉无用测试 |
| **Flaky 测试** | 随机失败 | 隔离 + 修时序 |
| **耦合测试** | 顺序依赖 | 数据隔离 |
| **脆弱选择器** | CSS 一改就坏 | 用 `data-testid` |

### CI/CD 集成示例

来自原文档的 GitHub Actions 配置：

```yaml
name: Test Pipeline
on: [push, pull_request]

jobs:
  unit-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run test:unit -- --coverage
        timeout-minutes: 5
      - uses: codecov/codecov-action@v3

  integration-tests:
    needs: unit-tests          # 依赖 unit 通过
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:15
    steps:
      - run: npm run test:integration
        timeout-minutes: 10

  e2e-tests:
    needs: integration-tests
    runs-on: ubuntu-latest
    steps:
      - run: npx playwright test
        timeout-minutes: 15
```

**C++/gtest 版本**:

```yaml
name: C++ Test Pipeline
on: [push, pull_request]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Configure
        run: cmake -B build -DCMAKE_BUILD_TYPE=Debug -DCODE_COVERAGE=ON
      - name: Build
        run: cmake --build build -j$(nproc)
      - name: Cache build
        uses: actions/cache/save@v4
        with:
          path: build
          key: build-${{ github.sha }}

  unit-tests:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/cache/restore@v4
        with:
          path: build
          key: build-${{ github.sha }}
      - name: Run unit tests
        run: ctest --test-dir build -L unit -j$(nproc) --output-on-failure
        timeout-minutes: 5
      - name: Generate coverage
        run: |
          lcov --capture --directory build --output-file coverage.info
          lcov --remove coverage.info '/usr/*' '*/test/*' --output-file coverage.info
      - uses: codecov/codecov-action@v4
        with:
          files: coverage.info

  integration-tests:
    needs: unit-tests
    runs-on: ubuntu-latest
    steps:
      - run: ctest --test-dir build -L integration --output-on-failure
        timeout-minutes: 10
```

### 关键设计模式

| 模式 | TS/JS 实现 | C++ 等价 |
|------|----------|---------|
| **测试金字塔** | 70/20/10 | 同样适用 |
| **页面对象** (Web E2E) | Page Object | 不适用 (C++ 多半不做 Web E2E) |
| **构建器模式** | TestDataBuilder | `Order::Builder().WithItem(...).Build()` |
| **工厂模式** | createUser() | `TestFactory::CreateUser()` |
| **测试夹具** | beforeEach | gtest `SetUp()/TearDown()` |
| **测试套件** | describe | gtest `TEST_F`/`TEST_P` |

### Flaky 测试管理

```typescript
// 隔离 flaky 测试 (TS)
describe.skip('Quarantined - INC-123', () => {
  test('flaky test awaiting fix', () => { /* ... */ });
});
```

**C++/gtest 等价**:

```cpp
// 方式 1: 用 DISABLED_ 前缀
TEST(MyTest, DISABLED_FlakyTest) {  // INC-123
  // ...
}

// 方式 2: 通过 GTEST_SKIP() 跳过
TEST(MyTest, ConditionalSkip) {
  if (IsFlaky()) {
    GTEST_SKIP() << "Quarantined: INC-123";
  }
  // ...
}

// 方式 3: filter 跳过
// ./my_test --gtest_filter=-*Flaky*
```

⚠️ **2 周 SLA 原则**：隔离的测试必须 2 周内修复或删除，否则就是技术债。

## 测试结果结构

agentic-qe 标准化的执行结果接口：

```typescript
interface ExecutionResults {
  summary: {
    total: number;
    passed: number;
    failed: number;
    skipped: number;
    flaky: number;     // 重试后通过的
    duration: number;
  };
  shards: ShardResult[];     // 分片结果
  failures: TestFailure[];   // 失败明细
  flakyTests: FlakyTest[];   // 不稳定测试
  coverage: CoverageReport;
  timing: TimingAnalysis;    // 性能分析
}
```

**gtest 输出 → 这个结构的映射**:

```bash
# gtest XML 报告
./my_test --gtest_output=xml:results.xml

# 字段对应:
# <testsuites tests="100" failures="2" disabled="1" time="12.5">
#   summary.total = tests
#   summary.failed = failures  
#   summary.skipped = disabled
#   summary.duration = time
#
# <testcase name="..." status="run" result="completed" time="0.5">
#   每个就是一条 test 记录
```

可以写个简单的 Python 脚本把 gtest XML 转成上面的 JSON 结构，统一上报。

## ⚠️ 关键提醒（原文）

1. **完整套件可能 OOM** —— 不要在 container 里一次跑全套，要分批
2. **少而精的 agent (3-4) 优于多而泛 (6-8)** —— 在 prompt 里加 verification command
3. **模型更新会改变行为** —— 昨天遵守的规则今天可能被忽略，要持续校准
4. **并行执行会掩盖 flaky** —— 初次诊断用 `--workers=1`
5. **Session 崩溃会丢上下文** —— 中间结果一定要写盘，不要只放内存

## 给 C++ 项目的自动化测试清单

按重要性排序：

1. ✅ **构建-测试分离**：`build` 和 `test` 是独立 CI job，复用构建产物（cache）
2. ✅ **分级执行**：unit (label: unit) → integration (label: integration) → e2e
3. ✅ **失败快速反馈**：`--output-on-failure` 立即看到失败原因
4. ✅ **并行**：`ctest -j$(nproc)`
5. ✅ **覆盖率上报**：lcov → codecov/sonarqube
6. ✅ **Flaky 隔离机制**：DISABLED_ 前缀 + 跟踪 issue
7. ⚠️ **智能测试选择**：基于 git diff 和 CMake 依赖图（需要自己实现）
8. ⚠️ **多平台矩阵**：Linux/Windows/macOS × GCC/Clang/MSVC
9. 💡 **变异测试**：用 mull-cxx 或类似工具验证测试质量
