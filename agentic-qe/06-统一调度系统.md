## 两层调度架构

```
┌─────────────────────────────────────────┐
│      qe-queen-coordinator               │  ← 战略层：拆任务 + 决策
└──────────────────┬──────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────┐
│      qe-fleet-commander                 │  ← 战术层：管 agent + 扩缩容
└──────────────────┬──────────────────────┘
                   │
       ┌───────────┼───────────┬───────────┐
       ▼           ▼           ▼           ▼
   Test Gen   Coverage    Security    Quality
   Domain     Domain      Domain      Domain
```

## Queen Coordinator（战略调度）

### 强制执行的 10 阶段协议（原文）

Queen 收到任何编排请求都必须按这 10 步走：

```
Phase 1:  aqe fleet init --json           初始化 fleet
Phase 2:  aqe health --json               健康检查
Phase 3:  aqe memory search ...           查询历史模式
Phase 4:  分析任务，选择 domains          决策
Phase 5:  aqe agent spawn "domain"        spawn 各 domain agent
Phase 6:  aqe task submit                 提交编排
Phase 7:  aqe task list (轮询)            等待完成
Phase 8:  aqe fleet status                收集指标
Phase 9:  aqe memory store                存储学习
Phase 10: 输出总结表格
```

### 任务路由表（自动匹配）

| 用户意图 | Queen 调用的 domain |
|---------|---------------------|
| "test coverage", "find gaps" | coverage-analysis |
| "generate tests", "write tests" | test-generation |
| "security scan", "audit" | security-compliance |
| "quality gate", "release ready" | quality-assessment |
| "run tests" | test-execution |
| "full QE", "comprehensive" | **所有 domain** |

### 禁止行为（来自原文 `<prohibited_behaviors>`）

```
❌ 永远不要只描述 agent 会做什么 —— 必须调用 MCP 工具
❌ 永远不要跳过 fleet_init —— 总是先初始化
❌ 永远不要跳过监控循环 —— 等任务完成
❌ 永远不要忘记存储学习
❌ 永远不要不跑工具就输出结果
```

## 依赖感知编排

### 依赖类型

| 类型 | 含义 | 调度行为 |
|------|------|---------|
| **hard** | 需要前置数据 | 先 spawn 前置，**等完成** |
| **soft** | 有更好，没也行 | 先 spawn 前置，**不等** |
| **peer** | 平级 | 同时 spawn |

| qe-root-cause-analyzer | — | qe-regression-analyzer, qe-defect-predictor |

### 分阶段 Spawn

```
完整发布验证:

Phase 1 (parallel):
  - qe-dependency-mapper
  - qe-coverage-specialist
  - qe-quality-gate
  - qe-risk-assessor

Phase 2 (parallel):     ← 等 Phase 1 完成
  - qe-impact-analyzer       (依赖 dependency-mapper)
  - qe-security-scanner      (依赖 dependency-mapper)
  - qe-gap-detector          (依赖 coverage-specialist)

  - qe-deployment-advisor
  - qe-root-cause-analyzer
```

**算法本质**: 拓扑排序 + 分层并行。

## 给 C++ 项目的轻量版调度器

如果你想自己搭，最小实现 (Python + SQLite ~100 行)：

```python
import json, sqlite3, subprocess
from concurrent.futures import ThreadPoolExecutor

class TaskScheduler:
    def __init__(self, db_path='.aqe/tasks.db'):
        self.conn = sqlite3.connect(db_path)
        self.conn.execute('''
          CREATE TABLE IF NOT EXISTS tasks (
            id TEXT PRIMARY KEY,
            agent TEXT,
            prompt TEXT,
            depends_on TEXT,
            status TEXT DEFAULT 'pending',
            result TEXT
          )''')
    
    def submit(self, task_id, agent, prompt, depends_on=None):
        self.conn.execute(
          'INSERT INTO tasks VALUES (?,?,?,?,?,NULL)',
          (task_id, agent, prompt, json.dumps(depends_on or []), 'pending'))
        self.conn.commit()
    
    def _is_done(self, task_id):
        row = self.conn.execute(
          "SELECT status FROM tasks WHERE id=?", (task_id,)).fetchone()
        return row and row[0] == 'done'
    
    def ready_tasks(self):
        ready = []
        for row in self.conn.execute(
          "SELECT id,agent,prompt,depends_on FROM tasks WHERE status='pending'"):
            deps = json.loads(row[3])
            if all(self._is_done(d) for d in deps):
                ready.append(row)
        return ready
    
    def execute_phase(self, max_parallel=4):
        ready = self.ready_tasks()
        with ThreadPoolExecutor(max_workers=max_parallel) as pool:
            list(pool.map(self._run, ready))
    
    def _run(self, task):
        task_id, agent, prompt, _ = task
        # 调用 AI CLI（举例：claude / codex）
        result = subprocess.run(
          ['claude', '-p', f'[{agent}] {prompt}'],
          capture_output=True, text=True)
        self.conn.execute(
          "UPDATE tasks SET status='done', result=? WHERE id=?",
          (result.stdout, task_id))
        self.conn.commit()
    
    def has_pending(self):
        return self.conn.execute(
          "SELECT COUNT(*) FROM tasks WHERE status='pending'").fetchone()[0] > 0


# 使用
s = TaskScheduler()
s.submit('gen', 'qe-test-architect', '生成 OrderService 测试')
s.submit('cov', 'qe-coverage-specialist', '分析 lcov.info')
s.submit('fill', 'qe-test-architect', '补充覆盖缺口',
         depends_on=['gen', 'cov'])

while s.has_pending():
    s.execute_phase(max_parallel=4)
```

## 调度系统关键决策

| 决策 | 选项 | 选择依据 |
|------|------|---------|
| 协调器层数 | 单层/双层/多层 | <10 agent 单层；>30 双层 |
| 通信方式 | 同步RPC/异步消息 | 异步消息几乎总更好 |
| 状态存储 | 内存/文件/DB | 至少文件，最好 DB |
| 失败处理 | 立即停/重试/跳过 | 测试可跳过；部署要停 |
| 扩缩容 | 手动/队列/预测 | 起步用队列阈值 |

## 实操建议

1. **先把单 agent 用顺** —— 不要一上来就做调度
2. **多 agent 时** —— 100 行 Python 调度器就够
3. **别学 agentic-qe 搞 60 个 agent** —— 10 个以内最可控
4. **可观测性优先** —— 必须有日志和状态查询接口
5. **失败要可重放** —— 任务定义和输入持久化
