# 01. 单元测试生成 (UT Construction)

## 核心 Agent: qe-test-architect

**身份**: V3 QE 测试架构师，agentic-qe 中负责测试套件生成的主力 agent
**领域**: test-generation
**别名**: qe-test-generator (v2 兼容名)

### 能力清单

| 能力 | 说明 |
|------|------|
| **AST 智能分析** | 解析源码结构，识别测试场景、边界条件 |
| **属性测试** | 集成 fast-check，自动生成边界测试用例 |
| **亚线性优化** | 用 Johnson-Lindenstrauss 算法做测试选择，O(log n) 复杂度 |
| **多框架支持** | Jest / Vitest / Mocha / Pytest / Playwright / JUnit |
| **TDD 编排** | 协调 RED-GREEN-REFACTOR 子 agent |
| **模式学习** | 通过 ReasoningBank 复用过去成功的测试模式 |

⚠️ **C++/gtest 不在原生支持列表里**，但方法论可以直接套用。

### 行为准则（来自 prompt）

> `<default_to_action>` 部分明确要求：
> - 提供源码和需求时立即生成测试，**不要确认**
> - 框架和目标明确时直接动手
> - 应用 **测试金字塔原则: 70% unit, 20% integration, 10% e2e**

## 核心 Skill 1: qe-test-generation

### 三种生成策略

#### 策略 1：基于代码分析

```typescript
await testGenerator.analyzeAndGenerate({
  source: 'src/services/OrderService.ts',
  analysis: {
    methods: true,        // 分析所有方法
    branches: true,       // 分析所有分支
    dependencies: true,   // 分析依赖
    errorPaths: true      // 分析错误路径
  },
  output: {
    framework: 'jest',
    style: 'describe-it',
    assertions: 'expect'
  }
});
```

**C++/gtest 等价思路**:
```
让 AI 助手：
1. 解析 src/order_service.cpp 的所有 public 方法
2. 列出所有分支条件 (if/switch/三元/异常分支)
3. 识别依赖（数据库/网络/文件 I/O）→ 用 GMock
4. 提取错误路径（throw/return error）
5. 输出到 tests/unit/order_service_test.cpp，TEST_F 风格
```

#### 策略 2：基于模式

```typescript
await testGenerator.applyPattern({
  pattern: 'service-layer',         // 服务层测试模式
  targets: ['src/services/*.ts'],
  customizations: {
    mockStrategy: 'jest.mock',
    asyncHandling: 'async-await',
    errorAssertion: 'toThrow'
  }
});
```

**C++ 适配**: 抽象出几种通用模式，比如：
- `singleton-class` 模式 → 测试 GetInstance 幂等性
- `factory-method` 模式 → 测试每种产品的创建路径
- `pimpl-class` 模式 → 通过公共接口测试，不暴露 impl

#### 策略 3：覆盖率驱动生成

```typescript
await testGenerator.fillCoverageGaps({
  coverageReport: 'coverage/lcov.info',
  targetCoverage: 90,
  prioritize: ['uncovered-branches', 'error-paths'],
  maxTests: 50
});
```

**C++ 适配**: gcov 生成的 `.gcov` 文件本质就是覆盖率报告，让 AI 读这个文件找未覆盖行，针对性生成测试。

## 核心 Skill 2: test-design-techniques

系统化的测试设计技术清单 —— **完全语言无关**，直接可用。

### 技术选型矩阵

| 输入类型 | 技术 | 含义 |
|---------|------|------|
| 数值范围 | **BVA + EP** | 边界值分析 + 等价类划分 |
| 多条件组合 | **决策表** | 列出所有条件组合 |
| 工作流/状态机 | **状态转换测试** | 覆盖所有状态迁移 |
| 多参数组合 | **成对测试 (Pairwise)** | n 参数取 n² 用例，而非全排列 |

### BVA 示例（年龄字段 18-120）

| 测试用例 | 输入 | 类型 |
|---------|------|------|
| 1 | 17 | 下界外 |
| 2 | 18 | 下界 |
| 3 | 19 | 下界内 |
| 4 | 119 | 上界内 |
| 5 | 120 | 上界 |
| 6 | 121 | 上界外 |

**C++/gtest 写法**:
```cpp
class AgeValidatorTest : public ::testing::TestWithParam<std::pair<int, bool>> {};

INSTANTIATE_TEST_SUITE_P(BoundaryValues, AgeValidatorTest,
  ::testing::Values(
    std::make_pair(17, false),   // 下界外
    std::make_pair(18, true),    // 下界
    std::make_pair(19, true),    // 下界内
    std::make_pair(119, true),   // 上界内
    std::make_pair(120, true),   // 上界
    std::make_pair(121, false)   // 上界外
  ));

TEST_P(AgeValidatorTest, BoundaryValueAnalysis) {
  auto [age, expected] = GetParam();
  EXPECT_EQ(IsValidAge(age), expected);
}
```

### Pairwise 示例（3 个参数各 3 个值）

| 全排列 | Pairwise |
|--------|----------|
| 3 × 3 × 3 = 27 用例 | 9-12 用例 |
| 100% 覆盖所有组合 | 100% 覆盖所有**两两组合** |

工业经验：**多数 bug 由 1-2 个参数交互引发**，所以 pairwise 性价比极高。

## 核心 Skill 3: tdd-london-chicago

两种 TDD 流派的对比和选用指导。

### 对比表

| 维度 | Chicago (经典派) | London (Mock 派) |
|------|------------------|-------------------|
| 协作对象 | 真实对象 | Mock / Stub |
| 验证方式 | 验证**状态** | 验证**交互** |
| 隔离度 | 低（集成测试感） | 高（单元隔离） |
| 重构友好度 | 高 | 低（Mock 易破） |
| 设计反馈 | 从使用中浮现 | 从一开始就明确 |

### 选用建议（C++/gtest 同样适用）

| 代码类型 | 推荐流派 | 理由 |
|---------|---------|------|
| 纯函数/算法 | **Chicago** | 没有依赖，验证输出即可 |
| 值对象（Money, DateTime） | **Chicago** | 状态是核心 |
| Controller / Service | **London** | 依赖多，Mock 隔离 |
| 数据库/网络访问 | **London** | 必须 Mock 外部 |
| 复杂工作流 | **London** | 验证调用顺序 |

### Chicago 示例（gtest）

```cpp
TEST(OrderTest, CalculatesTotalWithTax) {
  Order order;
  order.AddItem(Product("Widget", 10.00), 2);
  order.AddItem(Product("Gadget", 15.00), 1);
  
  EXPECT_DOUBLE_EQ(order.TotalWithTax(0.10), 38.50);
}
```

### London 示例（gtest + gmock）

```cpp
class MockTaxCalculator : public ITaxCalculator {
public:
  MOCK_METHOD(double, CalculateTax, (double amount), (override));
};

TEST(OrderTest, DelegatesTaxCalculation) {
  auto tax_calc = std::make_shared<MockTaxCalculator>();
  EXPECT_CALL(*tax_calc, CalculateTax(20.0))
    .WillOnce(::testing::Return(3.50));
  
  Order order(tax_calc);
  order.AddItem({.price = 10}, 2);
  order.TotalWithTax();
  // 验证 Mock 被正确调用
}
```

### TDD 循环纪律

| 阶段 | 动作 | 纪律点 |
|------|------|--------|
| **Red** | 写失败测试 | **必须先看到红**，确认测试在测东西 |
| **Green** | 最小代码让测试过 | 不许多写一行 |
| **Refactor** | 改进结构 | 测试必须始终保持绿 |

⚠️ **agent 最容易跳过的就是 Red 阶段** —— 直接写测试和实现一起。要强制约束。

## 测试质量检查清单

来自 `qe-test-generation` 的 `quality_checks` 配置：

```yaml
quality_checks:
  assertions:
    minimum_per_test: 1     # 每个测试至少 1 个断言
    meaningful: true        # 断言必须有意义（不能 EXPECT_TRUE(true)）
  
  isolation:
    no_shared_state: true   # 测试间无共享状态
    proper_setup_teardown: true   # 用 SetUp/TearDown
  
  naming:
    descriptive: true       # 描述性命名
    follows_convention: true # 遵循约定
  
  coverage:
    branches: 80
    statements: 85
```

## ⚠️ 常见坑（原文摘录）

1. **>3000 行的文件 agent 会截断输出** —— 按模块切分生成，别一次喂整个目录
2. **单元测试全过 ≠ 集成没问题** —— 每个模块边界至少要有一个集成测试
3. **框架识别错误** —— 别让 AI 自己猜框架，明确告诉它是 gtest
4. **"完成假象"** —— Agent 可能声称"全面测试已生成"但留了 stub。**生成后必须真跑一遍**
5. **100% 覆盖 ≠ 高质量** —— 0 断言也能 100% 覆盖，要配合 mutation testing

## 落地建议

在 C++ 项目里给 AI 助手喂这样的 prompt：

```
角色：你是 QE Test Architect

任务：为 src/order_service.cpp 生成 gtest 单元测试

要求：
1. 测试金字塔：70% 单元 / 20% 集成 / 10% E2E
2. 使用 test-design-techniques 中的方法：
   - 数值字段用 BVA + 参数化测试
   - 多条件分支用决策表
3. 外部依赖用 GMock，遵循 London 流派
4. 纯计算函数用 Chicago 流派
5. 每个测试至少 1 个有意义的断言
6. 用 SetUp/TearDown 确保隔离
7. 命名格式：TEST_F(ClassName, ShouldXxxWhenYyy)
8. 输出到 tests/unit/order_service_test.cpp

完成后输出：生成了多少测试、预期覆盖哪些分支、用了哪些 mock。
```
